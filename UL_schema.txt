Broad Profiling Database Architecture

Scope and Objectives
The scope of this document is the design of a central database for capturing publicly available broad profiling data generated by both CIRI and outside groups. Broad profiling assays are defined as any assay/platform that simultaneously generates hundreds to thousands of individual measurements per sample well, such as transcriptomics (HTTr) and cell painting (HTPP). In other words, broad profiling data is inherently high-dimensional, which creates additional challenges that the current ToxCast pipeline/inVitroDB are not set up to handle.
Specific goals:
•	Create a central repository for existing HTTr/HTPP/ETAP data that:
o	Captures study and sample-level meta-data across all data sets
o	Captures high-level summary values (e.g. chemical-level tPODs)
o	Points to separate study-specific data resources for larger scale levels of data, e.g. counts, fold-changes, individual curve fit results
•	Focus database on conc/dose-response studies
•	Implement database in PostgreSQL 18, with staging in DuckDB
Key questions and proposed answers (for now):
1.	Should HTTr, HTPP, and other assay types be combined into a single central DB, or should there be one such central DB for each assay type?
o	At the current stage, proposing to centralize several key tables for all study types, but have some tables, e.g. the individual well-level metadata, that are separated into 3 distinct study types (“domains”), as detailed below. Further schema harmonization and potential merging of these domains can be explored later.
2.	What external studies should be included/accommodated? Should focus be primarily on screening studies performed at EPA, or other sources and study designs?
o	At the current stage, proposal to include additional sources, such as data generated by HealthCanada and Unilever, but to focus on studies suitable for concentration-response modeling. See details in data architecture overview below.
3.	How should study-specific metadata be handled?
o	Propose to capture this in an extensible way, but will need input from DB admins on best way to implement. Initial plan is to put all study specific fields into several JSON column types.
4.	Should the summary statistics collected in the central DB be predefined, or flexible/extensible?
o	Given that there are multiple proposed ways to analyze this type of data, a flexible/extensible approach is proposed.
Data Architecture Overview
Three Primary Data Domains
The current proposal is to split data into three primary domains:
1.	High-throughput Transcriptomic (HTTr) screening data
o	Would be entirely in vitro and run using multi-well plates
2.	High-throughput Phenotypic Profiling (HTPP) screening data
o	Primarily refers to Cell Painting data
o	Would be entirely in vitro and run in multi-well format
3.	Low/mid-throughput Toxicogenomic (TGx) data
o	Includes both in vitro and in vivo studies
o	Includes a wider range of assays, e.g. could include metabolomics, proteomics, etc.
o	Individual studies cover one or several chemicals, or otherwise do not fit into general study design assumed in the first two DBs
o	Prime examples would be ETAP studies and data from Gwinn, et al. 2020
A longer term goal will be the harmonization and potential merging of these three domains into a single data architecture, but for now will aim to keep these separate, at least at the level of study meta-data.
Centralized Architecture
The current proposal is that for all of the three primary data domains above there be a single centralized database containing the following:
•	Study-level metadata – (covering all domains)
o	Study type – one of: HTTr, HTPP, TGx (everything else)
o	Profiling platform used (e.g. specific TempO-seq probe set)
o	Species, cell types, culture conditions, exposure conditions tested
o	Related publications
•	Chemical sample-level mappings – simply maps chemical sample identifiers to the corresponding chemical ID(s) for all domains
o	Chemical Sample ID (must be unique across all studies)
o	ROCID (future)
o	Standard IDs, e.g. ROCID, DTXSID, CASRN
o	Preferred name
•	Chemical-level summary values (each determined by one or potentially multiple standardized methods) – covering all domains
o	Chemical ID (will initially by DTXSID, then changeover to ROCID)
o	Chemical Sample ID
o	Corresponding study ID
o	tPOD/global POD values
o	Number of active features
o	Overall metric(s) of effect size/magnitude of response
o	Link to a separate table of “chemical series”, i.e. the set of wells analyzed to derive this POD – this may need to be domain-specific
•	Chemical-level Active Feature data – split by domain (and for HTTr, split further into gene-level and signature-level)
o	Contains only the conc/dose-responsive “hits” at certain feature-levels as needed for Navigator UI
o	i.e. the proposed UI for HTTr data will need each individual signature-level “hit” for each chemical
•	Chemical-level Well Series data – split by domain
o	This will provide linkage between well-level metadata and other Chemical-level tables (summary values and active feature data)
•	Well-level metadata – standardized and extensible metadata for each sample profiled in each study (e.g. each well of a screening study assay plate)
o	This will be split by domain, so there will be a separate table for HTTr, one for HTPP, and another for other TGx studies
For each study, or set of related studies, a separate data collection (doesn’t necessarily need to be a managed database) will capture:
•	“Raw” data – e.g. information collected from FASTQ (HTTr) and image (HTPP) files, count tables (HTTr/TGx), unnormalized numeric feature-level data(HTPP) 
•	“Intermediate” data – anything that is used to derive the final summarization values, but that is still high-dimensional and therefore not well suited to the centralized DB.
o	Examples include normalized counts, estimated fold-changes, feature-level curve-fit results, signature scores
•	“Feature” data – meta-data pertaining to the individual features measured
o	Individual probes in the probe set(s) used for TempO-seq studies
o	Individual features captured for Cell Painting studies
o	Individual genes measured for RNA-seq and other ‘omics studies
o	Note that some higher-level information about the feature set (e.g. descriptor, number of features) should go in the study-level metadata in the central DB.
o	Could consider capturing feature-level data in the central DB as well, since many studies will share the same feature set? But in all cases, there will be multiple possible feature sets.
•	“Supporting” data – anything else that is study specific, but not part of the centralized DB schema.
o	A primary example would be “null chemicals” simulated for each study to calibrate false positive rate of activity calls.

Cross-Domain Data
This section details the tables that will capture data across all three domains.
tgx_study Table:
One row per study with high-level information. One major restriction is that each study can only use a single platform (e.g. a specific TempO-seq probe set; a specific HTPP feature extraction set; a specific gene set for RNA-seq alignment). When a study covers multiple platforms, it should be split into multiple studies for the purposes of this data architecture.
•	study_id [SERIAL, primary key] – automated primary key linked to most other tables
•	study_name [TEXT] – A unique string identifier for the study, e.g. “epa_httr_u2os_screen”; will use this to link to larger data objects on disk for now
•	title [TEXT] – A short phrase describing the study, e.g. “HTTr Screen of 1500 Chemicals in U-2 OS Cells”
•	desc [TEXT] – A longer narrative description summarizing the study design (this can be several paragraphs, generally would pull from similar field on GEO or paper abstract)
•	source [TEXT] – Where this study was performed, e.g. “EPA”, “ULRI”, “Unilever”, etc.  
•	study_type [ENUM] – one of: HTTr, HTPP, TGx; indicates which domain the study is in
•	in_vitro [BOOLEAN] – Boolean indicating if study was in vitro or in vivo
•	species [ENUM   ]  – What species the study was performed in, e.g. “human”, “rat”, etc.
•	platform_id [INTEGER, foreign key] – ID of the profiling platform/associated feature set used (references same column in tgx_platforms)
•	cell_types [TEXT ARRAY] – One or more cell types or tissues  used for primary screening in the study (does not include QC controls)
•	culture_cond [TEXT ARRAY] – List of one or more descriptors for the culture condition(s) tested (e.g. different media or +/-AIME)
•	exposure_cond [TEXT ARRAY] – List of one or more descriptors for the exposure condition(s) tested other than concentration (e.g. different vehicles; different exposure times) 
•	refs [TEXT ARRAY] – List of one or more DOIs to publications describing this study; will eventually link this to other CIRI database for tracking publications
•	ext [JSON, optional] – Additional meta-data about the study (e.g. could capture similar info as desc, but in a more structured way)
tgx_samples Table:
One row per chemical/environmental sample that was screened, with mappings to other standardized identifiers.
•	chsamp_id [SERIAL, primary key]
•	spid [TEXT, optional?] – Any existing sample ID from other sample tracking systems (e.g. the sample IDs used in ChemTrack/EPA screening studies)
•	roc_id [TEXT, optional until ROC DB is up] – To be filled in when ROC IDs become available, links to ROC DB
•	dtx [TEXT, optional] – DTXSID of the corresponding chemical, will deprecate once roc_id is available and defer to DTXSID mapping from ROC DB
•	casrn [TEXT, optional] – CAS number of the corresponding chemical, will deprecate once roc_id is available and defer to CAS mapping from ROC DB
•	chem_name [TEXT, optional] – Preferred name for the corresponding chemical, will deprecate once roc_id is available and defer to name mapping from ROC DB
•	is_env [BOOLEAN] – Indicates if this was an environmental sample, if True then roc_id/dtx holds some other generic place holder that will not map to ROC DB
•	is_mix [BOOLEAN] – Indicates if this sample was a mixture (for future use)
•	ext [JSON, optional] – Additional meta-data about this sample (e.g. for some HTTr/HTPP studies we also tracked bottle number)
tgx_pods Table:
Each row captures a global/summary POD for a specific chemical sample screened in a specific study. There can be multiple types of PODs calculated from the same data, but only PODs that are meant to be an overall estimate of POD (e.g. tPOD) for a chemical based on broad profiling data belong in this table. This table should not include BMD/BMC values for individual features like genes or gene set signatures (see domain specific tables for those results). Every row should have a single corresponding row in both tgx_series and tgx_metrics, and the combination of those two foreign keys can be considered the primary key for this table.
•	series_id [INTEGER, foreign key] – foreign key matching series_id in tgx_series, join to that table for study_id, chsamp_id
•	metric_id [INTEGER, foreign key] – foreign key matching metric_id in tgx_metrics, join to that table for metric details
•	pod [NUMERIC, optional] – POD value (double/numeric); can be NA/Null
•	active [BOOLEAN] – Boolean, indicating whether this POD is considered “active” based on global POD activity calls (e.g. comparing to “null” chemical data)
•	ext [JSON, optional] – JSON encoded field storing any additional meta-data associated with this particular POD value, e.g. for ETAP studies we would want to capture both which tissue and which gene set had the lowest overall BMD.
tgx_series Table:
Each row describes a unique series of wells/profiles used for concentration/dose-response modeling. The actual mapping from tgx_series to individual wells is handled in domain-specific bridge tables, e.g. httr_series_wells
•	series_id [SERIAL, primary key]
•	study_id [INTEGER, foreign key]– references study_id in tgx_study
•	chsamp_id [INTEGER]– what chemical/environmental sample does this conc-response series correspond to, references tgx_samples
•	trt_cond [TEXT, optional] – brief description/code for the condition of this particular series (e.g. “AIME+” or “AIME-”). Can be blank for studies with only one culture and exposure condition
•	exp_lower [NUMERIC] – the lowest exposure conc/dose in the series
•	exp_upper [NUMERIC] – the highest exposure conc/dose in the series
•	exp_unit [TEXT] – the unit of measurement for the exposure conc or dose, e.g. “uM”, “mg/kg/d”, “mg/ml”  
•	exp_n [SMALLINT] – the number of exposure conc/dose groups in the series
•	exp_all [NUMERIC ARRAY] – the list of all exposure conc/doses in the series
•	ctrl_type [TEXT] – the type of vehicle control used, e.g. “DMSO”
•	factors [TEXT ARRAY] – list of any additional factors aside from conc/dose that were used in the conc-response model, e.g. “plate_id” for most HTTr/HTPP screening studies

tgx_metrics Table:
One row for each type of global POD metric captured in tgx_pods, e.g. “5th Percentile Signature BMD” for HTTr screens; “Lowest Gene Set BMD” for ETAP studies; “Global Mahalanobis POD” for HTPP studies.
•	metric_id [SERIAL, primary key]
•	metric_name [TEXT] – should be a user defined string, e.g. “httr_sig_perc5” for 5th percentile of signature BMCs for HTTr data
•	title [TEXT] – brief phrase describing this metric type, e.g. “5th Percentile of Signature BMCs”
•	desc [TEXT] – longer narrative description of this POD metric, can be several paragraphs
•	refs [TEXT ARRAY] – list of one or more DOIs pointing to publications where this metric was introduced, refined, and/or validated. This should not be a list of all papers that used the metric, rather it should be the primary papers that you would cite when using it in a new paper.
•	sw_name [TEXT, optional] – specific software package used when generating this metric, e.g. “BMDExpress” or “httrpathway”. Can be Null when metric comes from custom internal code.
•	sw_ver [TEXT, optional] – version of the software used . Can be Null when metric comes from custom internal code.
•	sw_url [TEXT, optional] – URL to the software, e.g. on CRAN or GitHub, ideally points to the specific version of the software used. Can be Null when metric comes from custom internal code.
•	ext [JSON, optional] – JSON encoded field capturing any other information, e.g. specific parameters used to derive this POD metric
tgx_platforms Table:
Each row summarizes a distinct broad profiling “platform” used for one or more studies in the database. Note that each version of each TempO-seq probe set and attenuation set is considered a distinct “platform” for the purposes of this database. Similarly, the catalog of genes/genome build used for quantifying RNA-seq, or the specific feature extraction set for Cell Painting, are each considered distinct platforms as well. 
At least at the level of this centralized database, we will only surface values from a single platform per study. For example: we might explore doing feature extraction with Harmony and then separately with Cell Profiler on some HTPP studies, but that level of data will only live in the study specific data object. Only one of these feature extraction methods and associated feature sets will be surfaced in the centralized DB architecture described here.
•	platform_id [SERIAL, primary key]
•	platform_name [TEXT] – short string identifying the platform type, e.g. humanWT2.1_brAtten
•	title [TEXT] – Short phrase briefly describing this platform, e.g. “TempO-seq Human WT v2.1, Broad Attenuation”
•	desc [TEXT] – Longer narrative description of this platform
•	ver [TEXT, optional] – Where appropriate, the specific version of the platform, e.g. for TempO-seq this would be the version of the probe set used; for RNA-seq it could be the version of the reference gene set used; for Cell Painting this could be 1 to start, but could increase if feature extraction software changed to include new features
•	tech_type [TEXT] – short string denoting the technology type, e.g. “TempO-seq”, “RNA-seq”, “Cell Painting”
•	study_type [ENUM] – One of “HTTr”, “HTPP”, or “TGx” denoting the domain where this platform is primarily used. To reduce redundancy here, this type does not have to match the type of all studies linked to this platform. Here it is used to point to the correct table of features associated with this platform. For example, some studies with type = “TGx” may use the same TempO-seq platforms as “HTTr” studies, and we don’t want to duplicate the feature information across 2 separate tables. In general, all TempO-seq platforms (and maybe other transcriptomic platforms) will have type = “HTTr” because the httr_features table will have the most appropriate schema for describing these types of platforms
•	species [ENUM, optional] – The target species for this platform; can be NA/Null for platforms that are not species specific, such as Cell Painting or metabolomics platforms
•	url [TEXT, optional] – When available, URL to the primary source of information for this platform, e.g. for TempO-seq, this would point to the specific probe manifest on BioSpyder website.
•	ext [JSON, optional] – JSON capturing additional fields for certain types of platforms, e.g. for TempO-seq this might be the attenuation type or version
HTTr Domain Data
Target Data Sets
In general, this data domain is meant to capture in vitro multi-conc high-throughput transcriptomic screening studies generated by EPA, ULRI, and other groups. List of potential data sets below:
•	All published EPA HTTr data sets, starting with data found on AWS Open Data
•	Additional in vitro studies published by Health Canada and Unilever (get links/references)
•	Any future in vitro HTTr data generated by ULRI/CIRI
httr_wells Table:
Each row captures meta-data for an individual well from a specific HTTr study, and the table encompasses wells from all HTTr studies. Note that this table is expected to be very large, initially hundreds of thousands of rows, eventually millions, but never billions of rows.
•	well_id [SERIAL, primary key] 
•	biosamp_name [TEXT] – A short string identifier for the biomaterial in the well. (generally this will be the name used on the sample key sent to contractor in the case of TempO-seq samples; when not specified in a study it can just be {plate}_{wellrow}{wellcol})
•	study_id [INTEGER, foreign key] – References tgx_study table
•	plate_id [TEXT] – A unique identifier for the plate, ideally these should be unique across all studies
•	well_row [ENUM, ‘A’ thru ‘P’] – The row for the well position on the plate
•	well_col [ENUM, ‘01’ thru ‘24’] – The column for the well position on the plate
•	pg_id [TEXT, optional] – The plate “group”, when part of the HTTr study design. Plates in the same pg_id are intended to be pooled together when performing conc-response analysis.
•	block_id [TEXT, optional] – The study “block”, when part of the HTTr study design.
•	cell_type [ENUM] – A standardized identifier for the cell type used, or for the specific type of reference standard
•	trt_name [TEXT] – A string uniquely identifying a specific type of treatment, e.g. “DEX_1uM” for 1 uM exposure of DEX. This is primarily meant to give a common label to replicate wells.
•	trt_cond [TEXT, optional] – A standardized string summarizing different treatment condition types within a study, e.g. ‘AIME+’ or ‘AIME-‘. This is primarily meant to differentiate conditions that are intended to be analyzed separately, especially for conc-response curve-fitting.
•	culture_batch [TEXT, optional] – An identifier for the cell culture “batch”, mainly needs to be specified when multiple plates were used to assay cells from the same batch
•	chsamp_id [INTEGER, foreign key] – Identifier for the chemical or environmental sample, references tgx_samples, can be Null for untreated wells
•	exp_time_h [SMALLINT] – The length of the exposure in hours
•	exp_conc [NUMERIC] – The concentration of the chemical sample used for the exposure
•	exp_veh [TEXT, optional] – The type of vehicle used for the exposure – when missing, can infer if there is only 1 type of vehicle control on the plate
•	qc_flag [ENUM] – Indication of whether the well was flagged with any QC issues, use a constrained set of values here (see previous HTTr Mongo Schema)
•	is_ref [BOOLEAN] Indicates whether this well corresponds to any type of reference sample, e.g. positive control, reference chemical treatment, or other QC standards
•	is_ctrl [BOOLEAN] Indicates whether this well corresponds to a vehicle control
•	is_trtd [BOOLEAN] Was the well treated in any way?
•	is_blank [BOOLEAN] Was this a blank well?
•	ext [JSON, optional] – Any additional fields related to study design, such as other relevant levels of sample batching, or additional exposure condition info.

httr_series_wells Table:
Bridge table linking rows in tgx_series to individual well meta-data in httr_wells. The combination of series_id and well_id should be unique for each row and can be treated as the primary key for this table.
•	series_id [INTEGER, foreign key] – references tgx_series
•	well_id [INTEGER, foreign key] – references httr_wells
•	is_ctrl [BOOLEAN] – was the specified well treated as one of the controls for the specified series
•	dose_level [SMALLINT] – Used to order the dose/conc groups from lowest to highest. Should always be 0 when is_ctrl is True. 

httr_features Table:
Each row captures a unique feature (e.g. TempO-seq probe) measured on one or more platforms in tgx_platforms. This table covers all platforms with type = “HTTr”. Note that the same feature can be part of multiple platforms (BioSpyder re-uses probes in different probe sets), and specific attenuation factors are captured in a separate linkage table to avoid duplicating too much information here
•	feature_id [SERIAL, primary key]
•	feature_name – Either the probe name (TempO-seq) or the preferred gene/transcript identifier for other platform types
•	probe_idnum [INTEGER, optional] – The integer part at the end of the probe name (TempO-seq only)
•	sequence [TEXT, optional ] – The probe sequence (TempO-seq), or the primary gene/transcript sequence for other types of platforms
•	transcripts [TEXT ARRAY] – A variable length list of all corresponding transcript IDs (can be a mix of different ID types)
•	gene_symbol [TEXT] – The official gene symbol of the target gene
•	entrez_id [TEXT, optional] – The gene ID in NCBI Entrez ID
•	ensembl_id [TEXT, optional] – The gene ID in Ensembl
•	probe_flag [ENUM] – marks probes that should be removed from further analysis after alignment. All probes that should be included in downstream analysis should be marked “OK”
•	ext [JSON, optional] – Any additional meta-data to keep track of for certain platforms
httr_platform_features Table:
This table serves as a bridge between tgx_platforms and httr_features, allowing for features to be part of multiple platforms with potentially different attenuation factors and alignment index ordering. Each row should have a unique pairing of platform_id and feature_id, and this combination of columns can be treated as the primary key.
•	platform_id [INTEGER, foreign key] – References platform_id in tgx_platform table
•	feature_id [INTEGER, foreign key] – References feature_id in httr_features table
•	index_order [INTEGER, optional] – The order this feature appears in the primary index for the platform, allows rebuilding alignment indexes directly from the database
•	attenuation [SMALLINT, optional] – The fold attenuation for this probe in the specific platform
httr_sig_cat Table:
One row per signature in the original gene set catalog, with associated meta-data needed for UI visualization.
•	sig_id [SERIAL, primary key]
•	sig_name [TEXT] – Text identifier for the signature, typically comes from the source catalog
•	Other fields will mirror the existing schema from HTTr study Mongo DBs, with focus on fields needed for surfacing signature-level results thru Navigator UI.
httr_sig_sets Table:
Each row summarizes a particular signature set (subset of the full catalog in httr_sig_cat) that was used for concentration-response analysis.
•	sig_set_id [SERIAL, primary key]
•	sig_set_name [TEXT] – Short identifier for a specific signature set, e.g. “screen_large”
•	desc [TEXT] – Description of the signature set
•	refs [TEXT ARRAY] – Identifiers to publications that used this specific signature set
•	replaces [TEXT, references sig_set_id] – Does this signature set replace an older one?
•	replaced_by [TEXT, references sig_set_id] – Was this signature set replaced by a newer one? 
httr_sig_set_cat Table:
Bridge table to link httr_sig_sets and httr_sig_cat tables. Each pair should be unique and can be treated as the primary key.
•	sig_set_id [INTEGER, foreign key] – references httr_sig_sets
•	sig_id [INTEGER, foreign key] – References httr_sig_cat
httr_sig_cr Table:
Each row is a distinct signature (gene set) concentration-response modeling result for a specific series in tgx_series. The combination of sig_id and series_id should be unique to a single row and can be treated as the primary key for this table. In the centralized database we would only store “active” (conc-responsive) signatures that are meant to be visualized in the Navigator UI, and only the fields necessary for that UI.
•	sig_id [INTEGER, foreign key] – references httr_sig_cat
•	series_id [INTEGER, foreign key] – references tgx_series
•	Other fields will mirror the existing schema from HTTr study Mongo DBs, with focus on fields needed for surfacing signature-level results thru Navigator UI.
